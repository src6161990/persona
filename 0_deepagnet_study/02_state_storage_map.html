상태 저장 지도

persona POC에서 누가 데이터를 만들고, 언제 저장이 트리거되며, 어떤 범위로 다시 읽히는지 연결한 지도입니다. 초록은 현재 실제로 연결된 구성, 보라는 현재는 연결되지 않은 StoreBackend 비교 대상입니다.

가장 중요한 한 줄 — Checkpointer는 Agent 실행 상태의 스냅샷을 저장하는 사람이고, StateBackend는 그 상태 안에서 Agent 파일을 다루는 방식입니다. StoreBackend는 별도의 LangGraph Store에 파일을 저장하는 방식입니다.
요청/트리거현재 POC에서 사용 중서비스 도메인 데이터현재 미연결 비교 대상

현재는 미연결. Agent 파일을 thread 밖에서도 꺼내야 할 때 별도로 연결한다. Checkpointer를 대체하지 않는다.
저장 트리거Agent가 그래프 단계를 진행하고 상태가 갱신될 때 Checkpointer가 checkpoint를 만든다. 캐릭터 편집 API를 다시 호출하면 같은 thread_id의 최신 상태가 이어진다.
StateBackend 파일Agent가 write_file() 같은 기본 파일 Tool을 쓸 때 생성된다. 그 파일은 Agent state에 있으므로 checkpoint 스냅샷에 함께 포함될 수 있다.
현재 핵심 변경save_character()는 backend 파일이 아니라 character_store를 직접 변경한다. 따라서 캐릭터 데이터는 Agent state와 별개다.

2. “저장소”별 경계와 열쇠(key)

개념 / 현재 여부저장하는 것누가·언제 쓰나범위와 다시 읽는 열쇠
Checkpointer
현재 사용		messages, tool 호출 진행, Agent state 스냅샷. StateBackend 파일도 state의 일부라면 같이 포함.LangGraph runtime이 Agent 실행 단계마다 저장. 앱은 checkpointer=CHECKPOINTER만 주입.thread_id = character-chat:{user_id}
같은 사용자의 캐릭터 편집 대화는 이어짐. 프로세스 재시작 시 InMemorySaver 내용은 사라짐.
StateBackend
기본값으로 사용		Agent가 보는 임시 가상 파일: /draft.txt, 내부 대용량 tool 결과 등.LLM이 Deep Agents 기본 file Tool을 호출할 때만 사용. 현재 커스텀 Tool 중심이라 파일을 적극 사용하지 않음.현재 Agent state / thread에 종속. 독립된 DB나 독립 조회 API가 아님. Checkpointer가 있어야 다음 요청에도 state를 복원할 수 있음.
StoreBackend
현재 미사용		LangGraph BaseStore 안의 Agent 파일. 체크포인트와 별개의 저장 경로.Agent file Tool이 읽기/쓰기할 때. create_deep_agent(backend=..., store=...)로 명시 연결해야 함.namespace로 user / tenant / assistant / thread 범위를 설계. 현재는 store가 없어 조회할 데이터도 없음.
app/store
현재 사용		정식 도메인 데이터: Persona, Character, Image, 통화 대화 이력.서비스 코드·Custom Tool이 저장. 예: save_character() → character_store.save().user_id 또는 call_id. Agent backend와 자동 연결되지 않음. 현재는 모두 인메모리라 재시작 시 사라짐.

3. multi-tenancy는 어디에 들어가는가?

현재 POC: user_id가 두 역할

URL의 user_id가 캐릭터/페르소나 조회 키이고, 서비스가 thread_id = character-chat:{user_id}를 만듭니다. 즉 같은 user_id의 편집 대화 상태가 이어집니다.

중요: 이는 “scope를 정한 것”이지, 인증·권한 검증 자체는 아닙니다.

멀티 테넌시가 필요해지는 순간

예를 들어 회사(tenant) A와 B가 같은 user_id 값을 가질 수 있거나, 한 사용자가 여러 workspace를 가질 때입니다.

그때는 (tenant_id, user_id, thread_id)처럼 각 저장소의 키/namespace를 분명히 설계합니다.

StoreBackend에서는

namespace=lambda rt: (tenant_id, user_id)처럼 namespace factory가 파일의 격리 범위를 만듭니다.

하지만 현재 POC는 StoreBackend를 연결하지 않았으므로, 이 규칙도 아직 실행 중이지 않습니다.

4. 내가 지금 조회할 수 있는 인터페이스는?

HTTP API 있음

정식 도메인 데이터

  • GET /personas/{user_id} — Persona
  • GET /personas/{user_id}/character — Character
  • 통화 이력은 현재 사용자 조회 API가 없음
UI / HTTP API 없음

Checkpointer 상태

  • 현재 InMemorySaver 객체는 프로세스 메모리에만 있음
  • LangGraph API의 get_tuple(config), list(config)로 개발 중 조회 가능
  • 운영자용 조회 화면은 아직 만들지 않음
현재 데이터 자체가 없음

State / Store backend 파일

  • StateBackend: standalone 조회소가 아니라 Agent state 일부
  • StoreBackend: 현재 미연결
  • 연결 후에는 Agent의 ls file Tool 또는 BaseStore API로 조회 가능

개발 중 checkpoint를 직접 들여다보는 최소 예시

이것은 학습/디버그용 Python 코드입니다. HTTP API로 노출하면 대화 내용이 유출될 수 있으므로 현재는 만들지 않습니다.

from app.agents.factory import CHECKPOINTER

config = {"configurable": {"thread_id": "character-chat:u1"}}
checkpoint = CHECKPOINTER.get_tuple(config)

if checkpoint is not None:
    print(checkpoint.checkpoint)  # 그 thread의 최신 Agent state 스냅샷
“StateBackend가 checkpoint에 저장된다”를 정확히 말하면?

아니요. StateBackend 자체가 checkpoint를 저장하는 것은 아닙니다. StateBackend가 Agent state 안에 파일 데이터를 두고, Checkpointer가 그 Agent state를 스냅샷으로 저장합니다. 반대로 StoreBackend 데이터는 별도 BaseStore에 있으므로 checkpoint 스냅샷의 주인이 아닙니다.

현재 앱을 재시작하면 무엇이 남나?

현재 CHECKPOINTER도 app/store 구현도 인메모리입니다. 따라서 Persona, Character, 통화 이력, 캐릭터 편집 상태 모두 프로세스 재시작 시 사라집니다. 이는 POC 학습 단계에서 의도적으로 단순화한 구성입니다.

이 지도에서 다음으로 실습해볼 가장 작은 실험은?

캐릭터 편집 Agent가 임시 메모를 write_file('/draft.txt', ...)에 쓰고 다음 턴에 읽도록 프롬프트와 가짜 모델 테스트를 추가해 보세요. 그러면 StateBackend → Checkpointer → 같은 thread_id 복원 연결을 실제로 관찰할 수 있습니다. 정식 Character 변경은 계속 save_character()에만 맡깁니다.
